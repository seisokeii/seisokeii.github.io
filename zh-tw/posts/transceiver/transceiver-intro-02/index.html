標準規範

在知道怎麼 debug 前需要先了解一下標準規範:

  • MSA
    • 定義尺寸、散熱、Pin 腳定義，就是硬體的部分啦。例如：QSFP-DD MSA, OSFP MSA
  • SFF-8024 (SFF Module Management Reference Code Tables)
    • 這個比較特別，它不是記憶體格式 (memory map)，而是一份由 SNIA 維護、給下面這些規範共用的「代碼字典」。
    • SFF-8472 / SFF-8636 / CMIS 在很多欄位都只存一個「代碼」，至於這代碼代表什麼意思（例如 Byte 0 的 Identifier 0x18=QSFP-DD、0x19=OSFP；或 Host/Media Interface code、Connector 類型…），都要回頭查 SFF-8024 的對照表。所以你會在那些文件裡一直看到「see SFF-8024 Table X」。
    • 它會持續改版（有新模組/新介面就加新代碼）。如果 NOS 內附的對照表比模組用的標準舊，就會把查不到的新代碼顯示成 Unknown（後面 SONiC 的 sff8024.py 會再提到）。
  • SFF-8472 (SFP/SFP+)
    • 將資訊切成兩個 I2C Address：0xA0 (基本資訊) 和 0xA2 (診斷資訊 DDM)
  • SFF-8636 (QSFP+/QSFP28)
    • 只有 Lower Page 00 (0-127 bytes) 是固定的
    • Upper Page (128-255 bytes) 需要透過寫入 Page Select Byte (Byte 127) 來切換
  • CMIS (Common Management Interface Specification) (QSFP56-DD/OSFP)
    • 以前 SFF-8636 是綁定 QSFP 外型的，但後來 OSFP 出來了，大家不想為每個形狀重寫 Spec 所以推出了 CMIS 這個規範
  • C-CMIS (Coherent CMIS)
    • 講白話就是一堆 threshold 的定義跟一些狀態紀錄，像是 OSNR, FEC, Laser Power, …之類的定義

這裡主要以 CMIS 來看就好，主要是因為目前手邊都在處理 400G/800G/1.6T 的模組測試。

不過不免俗的還是先從 SFF-8472 開始看起吧，畢竟了解歷史發展總是有助於理解為什麼現在的規範會變成如今這樣～

SFF-8472 (SFP/SFP+)

(玩硬體的好像都有自動上下顛倒能力…帕蒂瑪? 從0開始遞增但圖是從上到下)

上述這個圖可以看到 A0h 就是拿來定義一些這支 transceiver 的基本資料，而 A2h 就是 DDM (Digital Diagnostic Monitoring) 相關功能。

然後在 A2h 有很多個 128-255，也就是很多個 page 的意思 (這邊 Paging 是 Optional 功能)，可以看到 Page 00h/01h 的虛線部分，那 page 的切換是在 127 byte 那邊定義的。每個 page 也都有對應不同的功能用途。

根據 Spec 第 8.3 節 Table 8-3 ，A0h Byte 64 Bit 4 必須是 1，才代表這個模組支援 Paging。(之後 A2h Byte 127 用來選 page）。 （注意：同一個 Byte 64 的 Bit 3 是 Retimer/CDR indicator，不是 paging)

A2h Memory Map

懂了 A0h 後，那 A2h 其實大同小異～

在 transceiver 中會有 MCU 或是 DSP，會由這個晶片來負責監控這件事情(裡面有 firmware)，像是讀取溫度然後寫入到 A2h Page 00 Byte 96-97 中。

左大右小 (Big Endian)，96 MSB (Most Significant Byte) 與 97 LSB (Least Significant Byte)， 在溫度欄位這邊當作是整數跟小數醬子比較好理解～ (其他欄位的話就不一定是這樣子摟，通常是兩個 Byte 組成一個 16-bit unsigned integer)

在 SFF-8472 Ch3.3 提到: “Big Endian, that is the lowest byte address contains the most significant byte of the multi-byte value.”

Raw 格式的看法不難，看到最左邊的 column，那個是十六進位，拿黃匡舉例，60(0x60) 開始，往右看 hex 依序是 60(0x60),61(0x61),62( 0x62),…,6f(0x6f)，然後，去換算成十進位就可以知道是哪個 Byte 了。

可以看一下 Byte 96-97 (96 = 6*16^1 + 0*16^0)，也就是看左邊 column 的 60 開始，然後依序是顯示 20,47 (0x20,0x47)， 然後換算一下 2*16^3 + 0 + 4*16^1 + 7*16^0 = 8263，接著 SFF-8472 規定溫度的單位是 1/256°C， 8263/256=32.277°C， 那這邊會發現跟 SONiC 顯示的不一樣，主要是因為 SONiC 並不是隨時去讀取 EEPROM 的，而是定期去讀，然後儲存在 SONiC 自己的 DB 中，所以是有 delay 的。

再看個電壓(Vcc)例子，Byte 98-99 也就是接著顯示在溫度後面 82,53 (0x82,0x53)，換算完後是 33363， 然後 SFF-8472 規定電壓是 100 µV (0.1 mV)，所以 33363*0.1 mV ~= 3.336V 非常接近 SONiC EEPROM 顯示的數值了吧～

從上述例子來看，其實也是要去讀取規範文件中的單位來換算出對應的數值醬子，所以人工這樣看其實滿費力的。

SFF-8636 (QSFP+/QSFP28)

隨著速度提升到 40G/100G，模組需要紀錄的資訊變多了，但是 I2C Address 資源有限，不能一直由 A0, A2, A4… 這樣一直開下去。 於是 QSFP 系列的規範 (SFF-8636) 引入了一個很重要的概念：Paging (分頁)

文件中只有表格而沒有畫出立體圖，不過其實類似 SFF-8472 的 Memory Map，只是多的分頁功能。

這邊就不再長篇寫一堆了，畢竟上一篇說過簡單的到 100G 這邊而已，主要新增的概念就是規範 將 EEPROM 切分為兩半：

  1. Lower Page (0-127 bytes)：這是固定的，不管切到哪一頁，這邊的資訊永遠一樣（例如中斷狀態、監控摘要）
  2. Upper Page (128-255 bytes)：這是動態的

這時候 Byte 127 就變成了「電梯按鈕 (Page Select Byte)」。

常見的功能像是：

  • 寫入 00 -> Upper Page 顯示 Page 00h 的內容 (ID, Vendor Info)
  • 寫入 03 -> Upper Page 顯示 Page 03h 的內容 (Thresholds)

這個「固定下半部 + 切換上半部」的概念，是後來 CMIS 複雜架構的基礎。懂了這個，看 CMIS 就不會暈。

BTW. Vendor Name 跟一些基本固定資訊改到了 Page 00h，在 SFF-8472 是在 A0 讀取就好。

CMIS (QSFP56-DD/OSFP)

這邊呢要來囉唆一下，因為我在查資料的時候就感覺 spec 好像來源有很多 (SNIA, QSFP-DD MSA, OIF)，每次點下載都會覺得是不是錯誤網站在散佈惡意病毒…

  1. 為什麼 CMIS 5.0 在 QSFP-DD.com，而 5.2 在 OIF？
  • 因為 CMIS 的「監護權」移交了。
  • CMIS 2.x ~ 5.0 由 QSFP-DD MSA 撫養： 當初為了開發 400G，舊的 SFF-8636 不夠用，所以 QSFP-DD MSA (Multi-Source Agreement) 這個組織自己跳出來制定了 CMIS。當時這個規範是為了服務 QSFP-DD 這個硬體介面的。 所以 CMIS 2.0~5.0 都是在 qsfp-dd.com 下載
  • CMIS 5.1 以後交給 OIF 管理： 後來大家發現，CMIS 寫得太好了，連 OSFP MSA 和 COBO 也想用。既然大家都要用，繼續讓「QSFP-DD MSA」來管這個標準就很奇怪（感覺像 OSFP 在寄人籬下）。於是，業界決定把 CMIS 捐出來，交給一個更中立、更巨大的組織 —— OIF ( Optical Internetworking Forum) 來維護。
  1. SNIA, QSFP-DD MSA, OIF 三者關係圖
  • 這三個組織在光通訊模組上是「分層負責」的關係，缺一不可。可以把模組想像成 「一棟大樓」：
    • SNIA (SFF TWG) - 「圖書館管理員」 (基礎定義) SNIA 繼承了以前的 SFF 委員會。他們不負責最新的 400G/800G 邏輯，但他們掌管著 「字典」
      • 角色：維護所有模組共用的代碼表 (Code Tables)
      • 關鍵文件：SFF-8024
      • 不管你是 QSFP-DD 還是 OSFP，當你要在 Byte 0 填寫 “Identifier” 時，你要填什麼代碼？這要去查 SNIA 的書。CMIS 文件中會大量出現：「請參考 SNIA SFF-8024 表格 X」
    • QSFP-DD MSA - 「硬體建築師」 (硬體規範)
      • 這是一個由廠商組成的聯盟 (Cisco, Juniper, Molex 等)
      • 角色：專注於 QSFP-DD 的硬體尺寸、散熱、電的 Pin 腳 (Pinout)
      • 關鍵文件：QSFP-DD Hardware Specification
      • 現況：他們現在只管硬體殼要長怎樣，軟體部分就說：「請去遵守 OIF-CMIS」
    • OIF - 「物業管理」 (通用軟體規範)
      • 這是一個制定光傳輸協定的國際論壇
      • 角色：維護 CMIS (軟體介面)
      • 關鍵文件：OIF-CMIS-05.x
      • 目標：確保不管你的殼是 QSFP-DD 還是 OSFP，插進 Switch 後，軟體講的語言（I2C 讀寫規則、狀態機）都是同一套

搞清以上關係後… 對了解 CMIS 沒幫助呦呵呵~ 回歸正題。 來看一下 CMIS 5.0 的 Memory Map 圖：

看到這個圖好複雜怎麼連銀行都多出來了… 但沒關係先往下看

如果說 SFF-8636 只是透過分頁來擴充空間，那 CMIS 就是一次徹底的打掉重練。 主要原因是為了應付 QSFP-DD 和 OSFP 這種 8 通道 (8 Lanes) 的高速模組。

在 CMIS 中，我們不再把模組當成一個單純的「光電轉換器」，而是可以視為一個有自己 CPU、需要開機載入 Firmware 的「嵌入式系統」。

Why CMIS？

在 400G/800G 的時代，模組的功能變得很彈性。 一顆 400G DR4 的模組，它可能被 Switch 當作一個完整的 400G 使用，也可能被當作 4 個獨立的 100G 介面 (Breakout) 使用。 SFF-8636 那種固定的暫存器定義，已經無法描述這種動態的需求了。

Module State Machine

模組插上去沒 Link，就得去看 LOS (Loss of Signal)、LOL (Loss of Lock) 也就是是否有偵測到訊號，以及確認信號沒問題後鎖定。

CMIS 定義了嚴格的生命週期：

  1. ModuleLowPwr: 剛上電的低功率狀態，只開放 I2C 讓你讀 EEPROM 基本資料
  2. ModulePwrUp: 升到高功率、跑韌體初始化與自我校正
  3. ModuleReady: 模組就緒，等待 Host 設定 Application 之後就是 Data Path 層級的啟動：DPInit → DPInitialized → DPActivated，高速訊號才真正開始傳輸。

另外還有 ModuleFault 表示故障。完整狀態機請見 CMIS 6.3.2 Module State Machine。

如果模組卡在 ModulePwrUp，你換再多條光纖都沒用，因為它的 DSP 根本還沒進到 ready 狀態。

以上是建立在 EEPROM 以及 firmware 沒有任何錯誤的情況下…

Application Advertising

CMIS 有個很酷 (麻煩) 的發明，就是把「Switch 這邊的電訊號 (Host)」跟「光纖那邊的光訊號 (Media)」分開來管理。 中間透過 Application Advertising 來實現。

  • 模組會說：「我支援 A 模式 (800G) 和 B 模式 (2x400G)」
  • Switch 讀取後，寫入 ApSel (Application Select) 告訴模組：「我要用 B 模式 or A 模式」
  • 模組載入對應的 DSP 設定，Link 才會起來

這邊來看個例子:

首先可以看到在這個版本的 SONiC 的 show int trans eeprom 會寫目前每個 lane 是使用哪個模式(藍色框框)。
目前是看到 1 這個 1 就是去選擇 Application Advertisement 中的第一個，也就是 800GAUI。

那再看向上方 raw 格式，這邊要看兩個地方 10h,11h:

  • 10h 的 0x91~0x98 (RW) 這邊是指 Staged Control 也就是說 EEPROM 會告訴 SONiC 說 “我要求每條 lane 用哪個 App”, SONiC 要寫進去
  • 11h 的 0xCE~0xD5 (RO) 這邊是指 Active Control，就是每條 lane 正在使用哪個 App

SONiC 與 transceiver 的互動會是:

  1. Host 讀 Application Advertisement（Lower 0x56）→ 知道模組支援哪些 App
  2. Host 把想要的 ApSel 寫進「Staged」 Page 10h 0x91–0x98 ← 還沒生效，只是草稿
  3. Host 按下 ApplyDPInit（Page 10h 0x8F） ← 觸發「套用」
  4. 模組驗證 →（成功）把 Staged 複製到「Active」，並更新：
    • ConfigStatus Page 11h 0xCA–0xCD = 1(ConfigSuccess)
    • Active 設定 Page 11h 0xCE–0xD5 ← 現在才反映出來
  5. Data Path 初始化 → activated，link 起來

Memory Map 新增的：Banks

CMIS 有很多「每條 lane 各一份」的暫存器（每條 lane 的控制、狀態、監控…）。 一個 Page 的 lane-specific 區段最多排得下 8 條 lane；可是 QSFP-DD / OSFP 之後還有更多 lane 的模組（最多到 32 條），8 條就不夠分了。 於是 CMIS 在 Page 之上再加一層 Bank 的概念，你可以把它想像成「棟」。

  • Byte 126 (Bank Select): 選擇去哪一棟樓 (Bank 0 ~ Bank 3)
  • Byte 127 (Page Select): 選擇去哪一層樓

每個 Bank 對應「一組 8 條 lane」：Bank 0 = lane 1-8、Bank 1 = lane 9-16、Bank 2 = 17-24、Bank 3 = 25-32， 所以 banking 主要是給超過 8 lane的模組用的（最多 4 個 Bank = 32 lane）。像 8-lane 的 QSFP-DD / OSFP 只會用到 Bank 0（所以平常 dump 不太會碰到 bank 切換）。（C-CMIS coherent 模組另有「每條 media lane 一個 bank」的用法，不在這篇範圍。）這讓寫程式讀取 EEPROM 的複雜度又上升了一個維度。

VDM (Versatile Diagnostics Monitor)

VDM(Versatile Diagnostics Monitor,Page 20h–2Fh)是 CMIS 的「可自訂監控框架」。 跟溫度/Vcc 那種固定欄位不同,它是 descriptor 驅動:descriptor 描述「這格監的是什麼(OSNR / pre-FEC BER / 光功率 / Q…)」,值、門檻、告警旗標分散在不同 page(旗標在 Page 2Ch)。 coherent / ZR 模組的關鍵指標主要靠它。 因為要先讀 descriptor 才解得開,人工 raw 解很痛，可以自己去看文件XD

既然講到 ZR 這邊貼心提醒一下，如果有拿到 ZR/ZR+/CFP2 這種高功率的 transceiver 請先不要短接光纖，先插上去 switch 上面看 EEPROM。

就以這個圖片為例子，要先查看 RxPowerHighAlarm 是多少 dBm，然後再看另一端 transceiver 送出的 Supported Max TX Power 會不會大於 RxPowerHighAlarm，會的話請記得加衰減器，不然一隻 ZR 超過 4000 USD 就會燒毀呦～ 圖片上這隻我已經掛上了 -5 dBm 的衰減器了，對端是同型號的 TXPower 對端是 -7 dBm 左右，再加上 -5 dBm 衰減器，所以才會看到 -12 dBm 的 RxPower。
