Transceiver Initialization 流程分析

簡介

信息

本篇內容參考 CMIS 5.3（2026/5 發佈的 CMIS 5.4 為維護更新，暫存器佈局與本文相容， 故下述 offset 在 5.3 / 5.4 皆適用）。

OIF Official Documents

Appendix D Examples of Initialization and Deinitialization

讀完這篇文章可以了解：

  • 有哪些 init 流程（Transceiver 插入 Switch 之後）
  • Transceiver firmware 與 Switch NOS 如何溝通
警告

閱讀前請注意流程的「範疇」：

本篇描述的是 CMIS 規範定義的標準初始化流程（OIF-CMIS Appendix D 的範例）， 也就是「模組韌體與 Host 之間理想上該如何互動」的契約，採 Interrupt 驅動的握手模型 （Host 等待 IntL 中斷、讀取 Flag、再做下一步）。

實務上各家 NOS 的實作不一定逐字照做。以 SONiC 為例，它的 xcvrd 並不使用 IntL 中斷，而是用 polling（定時輪詢） 去讀 ModuleState / DataPathState / ConfigStatus 來推進狀態機。因此本文中「Host 偵測到 Interrupt」 這類步驟，在 SONiC 對應的是「下一輪輪詢讀到狀態改變」。

換句話說：本文講「規範上應該怎麼動」，不等於「某個 NOS 實際怎麼跑」。 兩者大方向一致，差別主要在觸發機制（中斷 vs 輪詢）。

大致上整個 Initialization 過程可以分成四個階段:

  • 發現與識別 (Discovery & Identification)
  • 管理層啟動 (Management Initialization)
  • 應用配置 (Application Configuration)
  • 資料通路啟動 (Data Path Activation)

行為可以分成兩種:

  • Host 端 → SONiC NOS 行為
  • Module 端 → Firmware 行為

第一階段：發現與識別 (Discovery & Identification)

提示

ModPrsL (Module Present Low) 是一根 實體針腳 (Hardware Pin)，

它不在 I2C 的 EEPROM 裡面。

  • QSFP-DD/QSFP 腳位定義：它是 Pin 27。

  • 物理行為 (Step 1 - Hot Plug)：

    1. 當模組還沒插進去時，Switch 主板上的上拉電阻 (Pull-up Resistor) 會把這根線拉到 High (3.3V)。
    2. 當您把模組插入 Cage (Step 1) 時，模組內部的這根 Pin 是直接接地 (GND) 的。
    3. 因此 Switch 主板上的電位瞬間被拉低變成 Low (0V)。

  • 軟體如何知道？

    • 通常 Switch 主板上有一個 CPLD 晶片連接著這根 Pin。
    • CPLD 會將這個電位變化轉換成一個 中斷訊號 (Interrupt) 通知 CPU。

  • 解釋 Step0 & 1

    • Step 0: Host applies Vcc 準備好電源 (Vcc) 並將 LowPwrRequestHW 拉高 (Assert)。
    • Step 1: Hot Plug。這就是 ModPrsL 被拉低的瞬間。

Step0: Host applies Vcc, asserts LowPwrRequestHW…

Host 啟用 Vcc 供電，並設定好 pin 預設值。

  • Host 端
    • 動作：Host 預設將硬體 pin LowPwrRequestHW 拉高 (Assert)。
    • 解讀：Host 雖然有給電 (Vcc)，但希望 Module 維持在 Low Power Mode。
  • Module 端
    • 無。

Step1: Hot Plug

當 Transceiver 插入。

  • Host 端
    • 動作：使用者將模組插入 Cage。
    • ModPrsL 狀態：從 High (1) 變為 Low (0)。
    • 解讀：Host 的 CPLD 偵測到這個 Low 訊號後，會觸發中斷通知 CPU：「有模組插入」。
  • Module 端
    • 無
提示
關於 ModPrsL pin： QSFP-DD / OSFP / QSFP 等 form factor 都有定義這根 Module Present 硬體接腳（QSFP-DD/QSFP 為 Pin 27），用來讓 Host 偵測模組是否插入， 與 I2C EEPROM 無關。模組插入時此 pin 接地被拉低，Host 端 CPLD 即可感知。
提示

補充說明：

Module 讀取 (I2C Read)

  • 動作：Host 軟體透過 I2C 讀取基本資訊。
  • 讀取資訊 (Register)：
    • Page 00h, Byte 0 (Identifier)：確認是 QSFP-DD (0x18) 或 OSFP (0x19)。
    • Page 00h, Byte 1 (Revision)：確認 CMIS 版本（例如 0x50 = 5.0）。
    • Page 00h, Byte 2 (MemoryModel)：
      • 檢查 Bit 7 (Memory Map)。
        • 如果是 0：Paged Memory (CMIS) ➔ 繼續流程。
        • 如果是 1：Flat Memory (Legacy) ➔ 不套用 CMIS 分頁初始化流程（見下方說明）

什麼是MemoryModel？

  • 0b 新模組 (CMIS)：有複雜的分頁機制 (Pages 00h-02h, 10h-11h supported…)。

  • 1b 舊模組 (Legacy SFF-8636)：沒有分頁 (Page) 的概念 (Page 00h supported only)。

提示

Flat Memory 為什麼不走這套流程？

CMIS 的分頁初始化流程（本文 Appendix D）是針對 Paged Memory 模組設計的， 仰賴 Upper Page 10h/11h 的控制與狀態暫存器來配置 Application、驅動 Data Path。

Flat Memory 模組（僅 Page 00h、無分頁，多為 SFF-8636 legacy 行為）沒有這些分頁暫存器， 因此 不適用 本文的 CMIS state machine。實務上 SONiC xcvrd 偵測到 flat_memory=1 時會直接把該 port 視為 READY、跳過 CMIS 狀態機（被動 DAC、部分 legacy 模組常見）。

所以這裡的「不套用 CMIS 分頁初始化流程」指的是：後續 Step 進入到分頁控制的部分對 Flat Memory 模組不成立，而非「模組壞掉」。

第二階段：管理層啟動 (Management Initialization)

Step2: Module powers up… sets LowPwrRequestSW=0, LowPwrAllowRequestHW=1…to Active Control Set and Staged Control Set 0.

Module 因 Vcc 啟動電源，但因為 Host LowPwrRequestSW 的訊號，所以載入一些相應設定。

將這些基本設定寫入到 Active Control Set, Staged Control Set 0。


LowPwrRequestSW = 0, (Page 00h, Byte 26, 4 bit)

LowPwrAllowRequestHW = 1, (Page 00h, Byte 26, 6 bit)

DPDeinit = 0, (Page 10h, Byte 128)

OutputDisableTx = 0, (Page 10h, Byte 130)

OutputSquelchForceTx = 0, (Page 10h, Byte 132)


現在模組為 M=MgmtInit, D=DPDeactivated 狀態。

  • Host 端
    • 動作：偵測到 Module presence 後，等待 Interrupt assertion (Flag)。
  • Module 端
    • 動作：Module 啟動電源，並載入預設設定。
    • 解讀：允許硬體控制 LowPwrAllowRequestHW 進入 Low Power Mode。
提示

什麼是 Staged Control Set 0-1, Active Control Set ?

簡單來說 CMIS 定義了「暫存區 Staged」「啟用區 Active」的分離設計。

  • 什麼是 Staged Control Set 0-1 ?
    • Host 在這裡填寫想要改變的參數（例如 AppSel Code、Lane Control …）。
    • 在這裡改的參數不會立即生效，直到所有參數設定好後才會複製到 Apply Active Set。
  • Staged Control Set 0 vs. Set 1？
    • 在 CMIS 文件中提到，它定義兩個設定暫存區。
    • Set0 為主要，Set1 為 optional (可能有些 Transceiver 不支援…)。
  • 什麼是 Active Control Set ?
    • 這是模組當下真正正在執行的設定，即模組當下真正的運作參數。
    • Host 不能直接修改這裡的數值。
    • 只能透過 Apply 指令，讓模組自己從 Staged 複製到 Active。
  • 什麼是 Provisioning?
    • 這是一個 「軟體層面」 的動作，重點在於 驗證 (Validation) 與 複製 (Copy)。
    • 定義：Host 觸發模組去檢查 Staged Control Set (草稿區) 的設定是否合法。如果合法，模組將這些設定複製到 Active Control Set (生效區)。
    • 發生時機：對應初始化流程的 Step 15 (ApplyDPInit) 到 Step 17 (ConfigStatus)。
  • 什麼是 Commissioning?
    • 這是一個 「硬體層面」 的動作，重點在於 落實 (Commit to Hardware)。
    • 定義：模組將 Active Control Set 中的設定真正應用到物理硬體 (Hardware) 上，使得硬體的實際行為與暫存器的設定一致。
    • 發生時機：
      • 通常緊接著 Provisioning 自動發生 (如果 SteppedConfigOnly=0)。
      • 對應初始化流程的 Step 19 ~ Step 20 (當 Data Path 進入 DPInit 狀態，並最終清除 DPInitPending 時，即 Data Path DPInitialized 之前的狀態)。

Step3: Module sets ModuleStateChangedFlag on entry into ModuleLowPwr

當 Module 確認進入 Low Power Mode 後改變 Flag 。


可以參考 Table 8-9 Module Flags:

ModuleStateChangedFlag (page 00h, Byte 8, bit 0)。


現在模組為 M=MgmtInit -> M=ModuleLowPwr, D=DPDeactivated 狀態。

  • Host 端
    • 動作：偵測到 Module presence 後，等待 Interrupt assertion (Flag)。
  • Module 端
    • 動作：進入 Low Power Mode 。
    • 解讀：更動 Flag 用於通知 Host，呼應 Host 行為 waits for Interrupt assertion 。
提示

有關 ModuleStateChangedFlag 變化可以參考以下表格：

  • 當 ModuleFault, ModuleLowPwr, ModuleReady 會讓 ModuleStateChangedFlag = 1。
  • 當 Flag 被 Assertion，Host 即可知道 ModuleState 狀態已改變。

Step4: Host detects assertion of Interrupt and reads all Flag…

偵測到來自 Module 的 interrupt 訊號 (assertion of Interrupt)。

  • Host 端

    • 動作 1：Host 讀取所有 Flag registers
    • 動作 2：並將 interrupt 訊號恢復 (deassert interrupt) ModuleStateChangedFlag (page 00h, Byte 8, bit 0) set 0b。
    • 解讀：確認 Module 符合預期行為，並進入下一個階段。

  • Module 端

    • 動作：無 (被動回應 I2C 讀取)。
    • 解讀：等待 Host 的電源讀取。

Step5: Host reads module power requirements

讀取 Module 電力需求。

  • Host 端
    • 動作：讀取 Page 00h, Byte 200 (ModulePowerClass)。
    • 解讀：檢查這顆模組需要多少瓦數 (例如 Class 7 = 14W)，並比對 Switch 本身的供電能力。
  • Module 端
    • 動作：無 (被動回應 I2C 讀取)。
    • 解讀：等待 Host 的電源讀取。
提示

ModuleStateChangedFlag 在哪裡讀？

它位於 Page 00h, Byte 8, Bit 0（Module Flags，見 Table 8-9）。當 ModuleState 變為 ModuleLowPwr / ModuleReady / ModuleFault 時，模組會把此 bit 設為 1 並（在規範模型下） 拉低 IntL 通知 Host。

Host 讀取此 Flag 的方式依 NOS 而定：規範模型是「收到中斷後讀 Flag」；SONiC xcvrd 則是用 polling 週期性讀取 ModuleState（Page 00h Byte 3）來判斷狀態是否前進， 而非依賴中斷與此 Flag。

Step6: Host writes FFh due to prevent automatic…

Host 寫入相關 register FFh，避免一當 ModuleReady 時會自動初始化 Data Path。

  • Host 端
    • 動作 1：Writes FFh to the DPDeinit register (Page 10h, Byte 128).
    • 動作 2：Writes FFh into the OutputDisableTx register (Page 10h, Byte 130).
    • 解讀：Host 先把 Module DPDeinit (強制資料通路重置) 和 OutputDisableTx (強制關閉雷射) 全部寫滿 1 (FFh)，限制模組行為；因為一當 ModuleReady Data Path 將自動被初始化，且 Data Path 進入到 DPInitialized 狀態，會自動啟用 Data Path activation 流程，所以需限制 Tx 狀態。
  • Module 端
    • 動作：接收 I2C 寫入指令，更新 Control Registers。
    • 解讀：保持 Data Path 關閉且雷射不發光。
提示

關於 OutputDisableTx（Page 10h, Byte 130）— 經 CMIS 5.3 Table 8-69 確認：

  • 位置就是 Byte 130（0x82），每個 bit 對應一條 media lane（bit0=media lane 1 … bit7=media lane 8）。
  • 0b = 該 media lane Tx 輸出開啟；1b = 關閉。
  • 重點：只有「實際存在的 media lane」那幾個 bit 有意義，超出 media_lane_count 的高位 bit 是 don’t-care，各家韌體可能填 0 或 1。

例：一支 coherent 800G ZR（media_lane_count = 1）讀到 0xFE，看似 7 條 lane 被關， 其實只有 bit0（唯一真實的 media lane）= 0 代表 Tx 開，bit1-7 對應不存在的 lane，可忽略。 判讀 Tx 是否被關時，務必先看 media_lane_count，只檢查存在的 lane。

Step7: clears LowPwrAllowRequestHW bit…

此時 M=ModuleLowPwr, D=DPDeactivated，Host 這邊也做好預防行為。


避免當 ModuleReady時，

Module 自動初始 Data Path，會最終將 Data Path 開啟，導致流程不受控制。

因此可知 Host 對於 Module 具有主導權，因此做好這些預防後，就把對 Module power lock 解除，也就是 LowPwrAllowRequestHW 訊號。

  • Host 端
    • 動作：Clears LowPwrAllowRequestHW bit (將 Page 00h, Byte 26, Bit 6 設為 0).
    • 解讀：強制模組忽略 hardware LowPwrRequestHW 訊號，直接進入 High Power Mode。
  • Module 端
    • 動作：接收 I2C 寫入指令 LowPwrAllowRequestHW 變為 0。
    • 解讀：更新自身 register 狀態。
提示

LowPwrAllowRequestHW

  • 0b: Module ignores the LowPwrRequestHW signal
  • 1b: Module evaluates the LowPwrRequestHW signal (default)

Step8: Module sees LowPwrS become FALSE and transitions to ModulePwrUp.

因為 Step7 Host 放開了對 Module 的電源控制，所以偵測到運作模式已改變。

  • Host 端
    • 動作：等待接收 Interrupt assertion 表示模組完成 High Power Mode 的流程。
    • 解讀：等待模組進入 ModuleReady。
  • Module 端
    • 動作：邏輯電路偵測到 LowPwrS 變成了 FALSE。
    • 解讀：模組將自身狀態從 ModuleLowPwr (001b) 切換到 ModulePwrUp (010b)。
提示

什麼是 LowPwrS？

LowPwrS 並不是一根外部的針腳，它是一個 模組內部的邏輯訊號 (Internal Logic Signal)。

CMIS 規範定義了一個邏輯閘來決定這個訊號。這個公式解釋了 Step 7 和 Step 8 的關聯：

參考：6.3.2.2 Module State Machine for Paged Memory Modules

在 Step7 之前，

因為 Module 初始化 (Step2) 設定自 registerLowPwrAllowRequestHW=1,LowPwrRequestSW=0:

LowPwrRequestHW (硬體 Pin) = 1,

LowPwrAllowRequestHW (軟體 Bit, Page 00h Byte 26: 6 bit) = 1

LowPwrRequestSW (軟體 Bit, Page 00h Byte 26: 4 bit) = 0

所以 LowPwrS = True。

但現在 Step7 之後因為 Host 控制 Module 的 LowPwrAllowRequestHW=0 bit,

LowPwrRequestHW (硬體 Pin) = 1,

LowPwrAllowRequestHW (軟體 Bit, Page 00h Byte 26: 6 bit) = 0

LowPwrRequestSW (軟體 Bit, Page 00h Byte 26: 4 bit) = 0

所以 LowPwrS = False。

Step9: Module powers up to High Power Mode.

啟動電源 Data Path 依然保持 DPDeactivated。


現在模組為 M=ModuleLowPwr -> M=ModulePwrUp, D=DPDeactivated 狀態。

  • Host 端

    • 動作：等待接收 Interrupt assertion 表示模組完成 High Power Mode 的流程。
    • 解讀：等待模組進入 ModuleReady。

  • Module 端

    • 動作：當**ModulePwrUp，**供電給高耗能元件（如 DSP、雷射驅動器 CDR、MCU 主核心）。
    • 解讀：執行自我檢測 (Self-Test) 與校正 (Calibration) 等等流程。

Step10: Module sets ModuleStateChangedFlag on entry into ModuleReady.

狀態由 ModulePwrUp -> ModuleReady，

Data Path 依然保持 DPDeactivated。


現在模組為 M=ModulePwrUp -> M=ModuleReady, D=DPDeactivated 狀態。

  • Host 端

    • 動作：等待接收 Interrupt assertion 表示模組完成 High Power Mode 的流程。
    • 解讀：等待模組進入 ModuleReady。

  • Module 端

    • 動作 1：將暫存器 Page 00h, Byte 3 的 3-1 bits 改為 ModuleReady (011b)。
    • 動作 2：將 Page 00h, Byte 8, Bit 0 (ModuleStateChangedFlag) 設為 1。
    • 動作 3：拉低外部硬體訊號 IntL (Interrupt Low)。
    • 解讀：當 **ModuleReady **通知 Host 可以開始進行 Stage 3 (Application Configuration)。

\

第三階段：應用配置 (Application Configuration)

Step11: Host detects assertion of Interrupt…

第二階段結束後，此時 M=ModuleReady, D=DPDeactivated，

因為 **ModuleReady**所以有送出 interrupt 到 Host。

等到 Host 接收到 interrupt 之後就會開始配置應用 (applying application)。

  • Host 端

    • 動作：Host 接收 Interrupt assertion ，表示模組完成 High Power Mode 的流程。
    • 解讀：模組進入 ModuleReady，所以 Host 可以開始做下一步流程。

  • Module 端

    • 動作：被動回應 I2C 讀取。

Step12: Host reads Application advertising registers

讀取模組支援的速率列表。

  • Host 端
    • 動作 1：讀取 Lower Memory (Page 00h), Byte 85，看 MediaType。
    • 動作 2：讀取 Lower Memory (Page 00h), Byte 86-117，看 AppSel 1-8。
    • 動作 3：讀取 Additional Application Descriptor Registers (Page 01h), Byte 223-250，看 AppSel 9-15。
    • 解讀：MediaType 可以看出是什麼類型的 Transceiver 介質，像是 MMF, SMF, AEC…；AppSel則可以看出模組的應用配置，像是 800GAUI-8, 400GAUI-4 等等…
  • Module 端
    • 動作：被動回應 I2C 讀取。
    • 解讀：提供自身的支援能力列表給 Host。
提示

當在讀取 I2C 有關 AppSel 這裡有很大一部分需要靠另外一個文件參照：

像是 HostInterfaceIDApp, MediaInterfaceID 對應的 Code 代表什麼意思，

比如說 HostInterfaceID=11h, MediaInterfaceID=1Ch就是對應到 400GAUI-8 C2M, 400GBASE-DR4。

SFF-8024

  • SFF Module Management Reference Code Tables, SNIA: SFF TA TWG (Storage Networking Industry Association: Small-Form-Factor Technology Affiliate Technical Working Group)

想要看到更多範例可以參考 CMIS Appendix C: Examples of Application Advertisements。

其實 SONiC 背後也有相對應的 script 在處理這件事情，也就是 sff-8024.py 負責將這些資訊轉意成 command: show int trans eeprom 看得懂的英文敘述。

補充：如果看到某一個 HostInterfaceID = FFh 代表他是最後一個 AppSel。

Step13: Host writes desired AppSel code into applicable Data Path Configuration Control registers in Staged Control Set 0

Host 瀏覽過所有 Module 支援的速率列表，並挑選一個適當的 application 套用。

通常 AppSel 1 是 Default。


將 AppSel 數據寫入到 Staged Control Set 0。

  • Host 端
    • 動作：將選定的 AppSel Code AppSelCodeLane 寫入到 (Page 10h), Byte 145-152。
    • \
  • Module 端
    • 動作：接收 I2C 寫入指令，將數值更新到 Page 10h 的暫存器中。
    • 解讀：僅先寫入到 Register 中，還沒被啟用，因為他是 Staged Control Set 0。
提示

什麼是 AppSelCodeLane ？

Step14: Host configures and enables host transmitters…

啟動 Host Tx 訊號，也就是 SerDes（Serializer/Deserializer，序列化/解序列化器）本身。

  • Host 端
    • 動作 1：Switch NOS 透過 SAI/SDK 下達指令給 Switch ASIC。
    • 動作 2：ASIC SerDes 被設定為目標速率 (e.g., 50G PAM4 per lane)。
    • 動作 3：ASIC 開始向模組發送高速電氣訊號 (Electrical Tx Enable)。
    • 解讀：建立物理層訊號 (Physical Signal)，讓模組端的接收器 (Rx CDR, Clock and Data Recovery 時脈資料恢復) 能夠鎖定時脈 (Lock)。
  • Module 端
    • 動作：物理層 (Host Electrical Interface) 接收到來自 Switch 的高速訊號。
    • 解讀：Module CDR 電路嘗試鎖定這個輸入訊號 (Input Signal)。

Step15: Host write a 1 to the bits in the ApplyDPInit register…

從 Step13 (Host 設定想要的 AppSel) → Step14 (Host 啟動 SerDes)。

現在 Step15 像是 commit config 的行為，

  • Host 端
    • 動作 1：對 Page 10h, Byte 143 (ApplyDPInitLane) 執行單次寫入 (Single Write)。
      • 如啟用 Lane 1~8：寫入 0xFF (1111 1111b)。
      • 如啟用 Lane 1~4：寫入 0x0F (0000 1111b)。(Lane 5-8 為 0, 即 Masked)。
    • 解讀：透過寫 0 來忽略不需配置的通道，確保只啟用預期的目標 Lane。
  • Module 端
    • 動作：接收 I2C 寫入指令，將數值更新到 Page 10h 的暫存器中。
    • 解讀：現在對於 Staged Control Set 0 的設定還沒結束。
提示

什麼是 ApplyDPInitLane ?

當寫入 1b 時，對應的 Lane 它會觸發 Provision procedure。

他會關聯到 ConfigStatusLane；

因為目前將設定寫在 Staged Control Set 0，等待 Active Control Set 應用。

什麼是 Single Write ?

是為了強調每一個 Lane applying 的同步性，舉個例子：若要啟用 Lane 1-8

  • 錯誤的做法
    • Host 先寫 Bit 0 (Lane 1)，再寫 Bit 1 (Lane 2)… 分 8 次寫。
    • Lane 1 先醒來 …依序… Lane 8 最後醒來。對於需要 8 個通道綁定在一起的訊號來說，這會導致 Skew (時間差)。
  • 正確的做法 (Single Write)
    • Host 送出一個 Byte (0xFF)。
    • Module 在同一個 CPU Clock Cycle 收到指令。
    • 8 個 Lane 同時開始初始化。

Step16: Module validates the configuration requested in Staged Control Set 0…

模組內部的 “Provisioning” (供應) 過程，檢查 Staged Control Set 格式正確及是否生效。

  • Host 端
    • 動作：等待 Module 跑完 Provisioning 流程。
  • Module 端
    • 動作 1：檢查 Staged Control Set 0 的參數有效性 (Validity Check)。
    • 動作 2：更新 DPInitPending 為 1b (Page 11h, Byte 235)。
    • 動作 3：更新 ConfigStatusLane 為 ConfigSuccess, 1h (Page 11h, Byte 202-205)。
    • 解讀：若 Validity Check 失敗，更新 ConfigStatusLane 為 ConfigRejected, 2h，則 Active Set 保持原狀。
提示
  • ConfigStatusLane
  • 什麼是DPInitPending?
  • 0b(Not Pending)
    • 解釋：模組初始化完成，現在硬體跟暫存器是同步的。
    • 發生時機：當模組真正執行完 Data Path 的初始化動作 (也就是 Step 20 結束時)，模組會自動把這個 Bit 清為 0。
  • 1b(Pending)
    • 解釋：Active Control Set (暫存器) 裡的內容已經是最新的了，但是 Hardware Configuration (實際模組) 還沒跟上，可能還在跑舊的設定。。
    • 發生時機：當 Step 15 (ApplyDPInit) 成功觸發 Provisioning 程序後，模組會瞬間把這個 Bit 設為 1。

Step17: Host reads the ConfigStatus fields to confirm …

Host 讀取有關 provisioning 的結果，看預期設定有沒有成功套用到 Active Control Set 上。

  • Host 端
    • 動作：讀取 Page 11h, Byte 202-205 (ConfigStatusLane)。
    • 解讀：檢查 provisioning 的結果，看是否有 ConfigSuccess。
  • Module 端
    • 動作：被動回應 I2C 讀取。
    • 解讀：回報 Staged Control Set 0 驗證與 Applying to Active Control Set 執行結果。

\

第四階段：資料通路啟動 (Data Path Activation)

Step18: Host requests initialization of the newly configured Data Path … clearing the DPDeinit bits…

在 Stage 2 (Step 6) 的時候，

Host 為了防止 ModuleReady 時自動初始化 Data Path DPInitialized 而將 Tx 訊號送出，所以刻意把 DPDeinit 設為 1 (強制鎖定)。


然而現階段完成 Stage 3 (Step 11-17) ，代表已經配置好 Data Path 設定到 Staged Control Set 0 上，也確認 ConfigStatusLane 為 ConfigSuccess 狀態，代表已經將 Host 預期的應用配置在 Active Control Set，使 Host 預期的 Module 的 Data Path 行為與實際相同。


最終 Host 將 **DPDeinit **鎖定清除，可讓 Module 進入到 DPInitialized 並載入 Host 預期的 Data Path 配置。

  • Host 端
    • 動作：將 Page 10h, Byte 128 (DPDeinitLane, Data Path Control) 的對應 Bit 清除為 0。
    • 解讀：Host 已經將 Data Path 設定好，所以將 Data Path 解除鎖定。
  • Module 端
    • 動作：接收 I2C 寫入指令，將數值更新到 DPDeinitLane (Page 10h, Byte 128) 的暫存器中。
    • 解讀：將 Data Path 解除鎖定。

Step19: Module sees DPDeinitS transition signal is FALSE and transitions the Data Path state to DPInit

因內部邏輯訊號改變，所以模組 Data Path 即將載入初始化程序。

  • Host 端
    • 動作：無。
    • 解讀：等待 Data Path initialization 完成後的 Interrupt assertion。
  • Module 端
    • 動作：Data Path 狀態機即將 (目前還沒) DPDeactivated 跳轉至 DPInit。
    • 解讀：內部邏輯 `DPDeinitS` 轉為 FALSE (因為 Step18 DPDeinitLane 被 Host 解除鎖定)。
提示

什麼是 DPDeinitS ?

它不是一個實體 pin，也不是 register 裡的值，是一個 模組內部的邏輯訊號 (Internal Logic Signal)。它與 Stage 2 (Step8) 的 LowPwrS 是相同性質的存在。

如果 DPDeinitS = 1 (TRUE)：強制資料通路關閉/重置 (Deactivated)。

如果 DPDeinitS = 0 (FALSE)：允許資料通路開始初始化 (Init)。→ 這是我們要的

使 DPDeinitS 為 FALSE 只有一個情境下符合：

  • 模組狀態是否為 ModuleReady。

  • 模組電源是否在 High Power Mode: LowPwrS = 0。

  • Host 是否有解除 Data Path 鎖定機制: DPDeinitLane = 0 (Page 10h, Byte 128)。

    • 在 Stage 2 (Step 6)，Host 把它設為 1 ➔ 鎖定。
    • 在 Stage 4 (Step 18)，Host 把它設為 0 ➔ 解鎖！

    \

Step20: Module initializes the Data Path and clears its DPInitPending bits.

模組執行內部初始化程序，並更新 DPInitPending 狀態。


由於 (Step 18) 已經將 DPDeinit 鎖定解除，所以此時現在的狀態是：

M=ModuleReady, D=DPDeactivated -> D=DPInit，代表現在正在初始化階段。

  • Host 端
    • 動作：等待 Data Path initialization (D=DPInitialized) 的 Interrupt assertion。
  • Module 端
    • 動作：清除 DPInitPending 為 0b (Page 11h, Byte 235)。
    • 解讀：當 DPInitPending 變為 0 時，代表 Active Control Set 的參數已經真正落實到模組電路上了。

Step21: Once the Data Path is initialized the module sets the DPStateChangedFlag bits, on entry into DPInitialized.

模組完成初始化後隨之進入 DPInitialized 狀態。

並更新 DPStateChangedFlag 即通知 Host 端程序已經完成。

因此現在的狀態為：

M=ModuleReady, D=DPInit -> D=DPInitialized。

  • Host 端
    • 動作：等待 Data Path initialization (D=DPInitialized) 的 Interrupt assertion。
  • Module 端
    • 動作：模組將 DPStateChangedFlag (Page 11h, Byte 134) 的對應 Lane Bit 設為 1。
    • 解讀：當正式進入 DPInitialized 狀態後，DPStateChangedFlag 因狀態改變需要設定為 1。
提示
  • 這個表格詳細描述了 DPStateChangedFlag 何時會被 Assertion。

Step22: Host detects assertion of Interrupt and reads all Flags to clear the interrupt.

檢查目前所有的 Flag 狀態，清除 interrupt。

此時 Host 讀取到 Module 已經進入到 DPInitialized，因此清除相關 Interrupt ，並準備進入到下一階段（將開始逐步啟用 Tx）。

  • Host 端
    • 動作：Host 接收 Interrupt assertion ，並清除 DPStateChangedFlag。
    • 解讀：模組進入 DPInitialized 的狀態，所以 Host 可以開始做下一步流程。
  • Module 端
    • 動作 1：被動回應 I2C 讀取 Flags 資訊。
    • 動作 2：接收 I2C 寫入指令，將 0b 更新到 DPStateChangedFlag (Page 11h, Byte 134)。

\

Step23: Host requests activation of the Data Path … Host writes the new contents of the OutputDisableTx register in a single write transaction.

Host 在能互相傳送流量前，需請求啟用 Data Path，設定 0b: clearing the OutputDisableTx。


回顧一下約在 Stage 2 (Step6) 時：

當時 Host 為了防止 Data Path 為 DPInitialized時，自動載入 Data Path 啟動程序(automatic Data Path activation)，因此將在 Step 6 時，Host 設定 OutputDisableTx 為 FFh。


現在因為 Host 想要啟用 Data Path，所以需要將這個欄位清零 0b。

  • Host 端
    • 動作：將 OutputDisableTx 設定為 0b (Page 10h, Byte 130)。
    • 解讀：因為 Host 想要啟用 Data Path，所以將先前做的預防鎖打開。
  • Module 端
    • 動作：接收 I2C 寫入指令 OutputDisableTx 變為 0。
    • 解讀：Host 允許 Module 自動執行 Data Path 啟動程序。
提示

在對 OutputDisableTx register 進行操作時要注意以下事項：

  • read-modify-write
    • 特別是在 Breakout 設定時需要注意，因為可能有些 Lane 已經啟用正在服務，舉例來說 OutputDisableTx = (1111 0000)b，這代表 Lane 1~4 正在使用，Lane 5~8 屬於關閉狀態，若 Host 現在想啟用 Lane 5~8，就需要有「先讀後寫」的機制，避免預期得結果是 (0000 0000)b，被誤寫成 (0000 1111)b，導致 Lane 1~4 無預期被停用。
  • single write transaction
    • Host 必須用 一個 I2C 指令 同時更新所有目標 Lane 的 Bit。

Step24: Module sees DPDeactivateS transition signal is FALSE and transitions the Data Path state to DPTxTurnOn.

內部邏輯訊號改變，Data Path 允許改變狀態至 DPTxTurnOn。

  • Host 端
    • 動作：等待 Module 完成 Data Path activation 時的 Interrupt assertion。
  • Module 端
    • 動作：因為模組運作邏輯 DPDeactivateS 變成了 FALSE，所以 Data Path 變成了 DPTxTurnOn。
提示

什麼是 DPDeactivateS ?

這是一個 模組內部的邏輯訊號 (Internal Logic Signal)。

如果 DPDeactivateS = 1 (TRUE)：無法完全啟動 (進入 DPTxTurnOn 或 DPActivated)。

如果 DPDeactivateS = 0 (FALSE)：允許資料通路啟用 (activated)。→ 這是我們要的

DPDeactivateS的定義是只要有任一個值為 TRUE，Data Path 就無法被啟用：

  • DPReDeinitS: 是否正在重新初始化。
  • DPTxDisableT: 是否 Tx 被 Host 關閉
  • DPTxForceSquelchT: 是否被 強制靜噪 (Squelch)。

什麼是DPTxForceSquelchT ？

它是由 OutputSquelchForceTx 組成的 8~1 分別對應到每一個 Lane，以下情況可能會使用到這項功能：

  • Switch 開啟靜噪 (Force Squelch)
    • Switch A 維持雷射開啟，但開啟 OutputSquelchForceTx。
    • Switch B 仍然收到光 (LOS = 0)。
    • 但是 Switch B 的 CDR (時脈資料恢復電路) 鎖不住訊號觸發 LOL (Loss of Lock)。
    • 所以 Switch B 知道硬體連線是好的，只是對面處於「故障/維護模式」。
  • 雷射熱穩定性 (Thermal Stability)
    • 這在 DWDM (高密度波分複用) 或 Coherent (同調光) 模組中可能會使用。
    • 使用 Squelch：電流繼續流過雷射 (保持發光)，且不處理訊號，讓雷射的溫度維持不變 (熱平衡)。
  • 調整 DSP 的參數時
    • Host 修改 Equalizer 設定，避免調整訊號時送出無效訊號。

Step25: Modules enables all Tx outputs for the Data Path being activated

這是 Tx Turn On 過程的物理執行階段。

現階段由 M=ModuleReady, D=DPInitialized → D=DPTxTurnOn。

  • Host 端
    • 動作：等待 Module 完成 Data Path activation 時的 Interrupt assertion。
  • Module 端
    • 動作 1：雷射驅動器 (Laser Driver) 開始供應偏壓電流 (Bias Current)，光發射次組件 (TOSA, Transmitter Optical Sub-Assembly) 開始發光。
    • 動作 2：模組內部的自動功率控制 (APC, Automatic Power Control) 迴路開始運作，等待雷射溫度穩定。
    • 動作 3：光訊號從光纖接口射出。
    • 解讀：Tx 完全開啟 的過程。

Step26: Once the Data Path is activated the module sets the DPStateChangedFlag bits, on entry into DPActivated

現階段由 M=ModuleReady, D=DPTxTurnOn → D=DPActivated。

標誌著 Data Path 正式進入運作狀態 (completion of DP activation)。

  • Host 端
    • 動作：等待 Module 完成 Data Path activation 時的 Interrupt assertion。
  • Module 端
    • 動作 1：完成所有 Data Path activation 後，正式進入 DPActivated。
    • 動作 2：將 DPActivated 狀態記錄到 DPStateHostLane register (Page 11h, Byte 128-131)。
    • 動作 3：因進入 DPActivated 所以將 DPStateChangedFlag 設為 1，通知 Host 中斷。
    • 解讀：以上模組動作完成後代表資料通路 DPActivated。

第五階段：傳輸流量 (Carry the traffic)

Step27: Host detects assertion of Interrupt …

Host 讀取到 DPStateChangedFlag Interrupt，發現 Module 已經進入到 DPActivated。

  • Host 端
    • 動作：Host 接收 Interrupt assertion ，並清除 DPStateChangedFlag。
    • 解讀：模組進入 DPActivated 的狀態，所以 Host 可以開始做下一步流程。
  • Module 端
    • 動作 1：被動回應 I2C 讀取 Flags 資訊。
    • 動作 2：接收 I2C 寫入指令，將 0b 更新到 DPStateChangedFlag (Page 11h, Byte 134)。

Step28: Host uses the activated Data Path to carry live traffic

Host 使用已啟動的資料通路來傳輸實際流量

收尾
